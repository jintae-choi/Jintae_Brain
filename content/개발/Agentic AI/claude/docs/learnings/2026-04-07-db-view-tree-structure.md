# WBS 정규화와 뷰를 활용한 역정규화 전략

- **날짜:** 2026-04-07
- **프로젝트:** bcmf_bs
- **분류:** 패턴 / DB 설계

## 문제/상황

### 기존 구조 (안티패턴)
기존에는 **하나의 테이블에 lv1~lv7 중 해당 레벨까지만 값을 채워서** 사용하고 있었다:

```
[기존 단일 테이블] — 안티패턴
wbs_cd                      name          lv1   lv2      lv3      lv4   lv5    lv6   lv7
W-001                       도로          도로
W-001001                    구조물공      도로  구조물공
W-001001001                 대호지교      도로  구조물공  대호지교
W-001001001001              양방          도로  구조물공  대호지교  양방
W-001001001001002           하부공        도로  구조물공  대호지교  양방  하부공
W-001001001001002001        교대          도로  구조물공  대호지교  양방  하부공  교대
W-001001001001002001001     기초&저판     도로  구조물공  대호지교  양방  하부공  교대  기초&저판
W-001001001001002001002     교대본체      도로  구조물공  대호지교  양방  하부공  교대  교대본체
W-001001001001002001003     뒷채움...     도로  구조물공  대호지교  양방  하부공  교대  뒷채움...
                                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                          상위 레벨 값이 하위 행마다 반복 → 데이터 중복
```

**문제점:**
- 위치(location)와 공종(work_type) 두 도메인이 한 테이블에 혼재
- 상위 레벨 값이 하위 행마다 반복 (정규화 위반)
- "교대" 이름이 바뀌면 해당 행 전체를 UPDATE 해야 함
- FK 참조 없이 문자열에 의존하여 데이터 무결성 보장 불가

### 정규화 후 구조 (현재)
원자성을 지키기 위해 3개 테이블로 분리했다:

```
tb_location (자기참조 트리 — Adjacency List)
┌─────────────┬───────────┬──────────┬───────┐
│ location_id │ parent_id │ name     │ level │
├─────────────┼───────────┼──────────┼───────┤
│ 1           │ NULL      │ 도로     │ 1     │
│ 2           │ 1         │ 대호지교 │ 2     │
│ 3           │ 2         │ 양방     │ 3     │
└─────────────┴───────────┴──────────┴───────┘

tb_work_type (자기참조 트리 — Adjacency List)
┌──────────────┬───────────┬──────────┬───────┐
│ work_type_id │ parent_id │ name     │ level │
├──────────────┼───────────┼──────────┼───────┤
│ 1            │ NULL      │ 구조물공 │ 1     │
│ 6            │ 1         │ 하부공   │ 2     │
│ 7            │ 6         │ 교대     │ 3     │
│ 8            │ 7         │ 기초&저판│ 4     │
└──────────────┴───────────┴──────────┴───────┘

tb_wbs_mapp (교차 테이블 — Junction Table)
┌────────┬─────────────┬──────────────┐
│ wbs_id │ location_id │ work_type_id │
├────────┼─────────────┼──────────────┤
│ 3      │ 3 (양방)    │ 8 (기초&저판)│
└────────┴─────────────┴──────────────┘
```

- "교대" 이름이 바뀌면 tb_work_type 1행만 수정
- 데이터 중복 없음, FK로 무결성 보장

### 해결해야 할 상황
팀원이 정규화된 현재 구조를 이해하지 못하고, 기존처럼 lv1~lv7이 펼쳐진 형태로 데이터를 봐야 한다고 요구했다. 이에 대한 해결책으로 **뷰(View)**를 만들었다:

- 실제 데이터는 정규화된 3개 테이블에 저장 (원자성 유지)
- 뷰(vw_wbs)가 조회 시 3개 테이블을 조인하여 기존과 동일한 형태로 보여줌
- **저장 구조는 정규화, 보여주는 형태는 역정규화** — 양쪽 요구를 모두 충족

### vw_wbs 뷰 최종 구조 (377행)

`source` 컬럼으로 각 행의 name이 어느 테이블에서 온 것인지 명시한다.

**구조물공 패턴** (location lv3 + work_type lv4 = 7)

W-001001001001002001001      7  기초&저판   work_type            3             8       3

**토공 패턴** (location lv4 + work_type lv3 = 7)

## 배운 것

### 데이터 모델링 패턴 용어
- **Adjacency List**: parent_id로 부모-자식 관계를 표현하는 트리 저장 방식. 가장 기본적이고 널리 쓰임
- **Junction Table (= Mapping Table, 교차 테이블)**: N:M 관계를 구현하기 위한 중간 테이블. ERD에서 "관계"는 테이블 간 연결 자체를 말하고, Junction Table은 그 관계를 구현하는 테이블
- **안티패턴(Anti-pattern)**: 자주 쓰이지만 나중에 문제가 생기는 설계. 반대말은 패턴(검증된 좋은 해결책)

### 뷰(View) vs 테이블
- **뷰는 저장된 쿼리**이므로 비정규화된 데이터(lv1~lv7 반복)를 담아도 원자성을 깨지 않음
- 같은 구조를 실제 테이블로 만들면 안티패턴 (서로 다른 도메인 혼재, FK 없이 문자열 의존)
- 뷰와 백엔드 로직은 경쟁이 아닌 역할 분담: 뷰는 DB 레벨 확인/분석용, 백엔드는 API 응답용

### sort_order 기반 계층 코드 설계
- `W-` + 각 레벨 sort_order 3자리 연결 → 문자열 정렬 = 트리 순서
- sort_order가 같은 부모 아래에서 유일하면 코드도 자동으로 유일
- 뷰에서 생성하므로 sort_order 변경 시 코드가 자동 반영 (동기화 문제 없음)

### source 컬럼의 필요성
- lv7은 location_id, work_type_id, wbs_id 세 개 다 있는데 name은 work_type에서 옴
- lv5는 행마다 location일 수도, work_type일 수도 있음
- ID 조합으로 출처를 추측하는 건 혼란스러우므로 `source` 컬럼으로 명시하는 게 정확

### UNION ALL + DISTINCT로 트리 전개
- 리프 노드에서 7개 UNION으로 lv1~lv7 각 레벨 행 생성
- DISTINCT로 공유되는 부모 노드 중복 제거
- 각 행에 원본 테이블 ID(location_id, work_type_id, wbs_id)를 추가해야 추적 가능

## 기억할 포인트
- 두 독립 트리를 하나로 합쳐 보여주는 건 **뷰로 만들면 OK, 테이블로 만들면 안티패턴**
- 계층 코드는 sort_order 3자리 연결이 가장 직관적 (정렬 = 트리 순서)
- 뷰 컬럼 구조가 바뀌면 `CREATE OR REPLACE`가 안 되고 `DROP → CREATE` 해야 함
- name 출처가 모호한 경우 `source` 컬럼을 추가하여 명시하는 게 실수 방지에 좋다
