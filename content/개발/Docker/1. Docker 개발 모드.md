# 개요

Quartz 기반 이 프로젝트는 로컬 실행을 **Docker로만** 한다. Node 직접 실행(`npx quartz build --serve`)은 금지. 휴대성 때문이다. 어떤 PC든 Dockerfile + docker-compose.yml만 있으면 동일한 환경이 재현된다.

문제는 작업 종류에 따라 **볼륨 마운트 구성이 달라야 한다**는 점이다. 콘텐츠만 수정할 땐 엔진 코드를 고정하고 싶고, 디자인·컴포넌트 수정할 땐 엔진 코드도 즉시 반영되어야 한다. 이를 해결하기 위해 **기본 모드 / dev 모드 2가지**로 분리했다.

# 전체 개념

## 기본 모드 (prod-like)

`docker-compose.yml`만 사용.

볼륨 마운트 대상:
- `./content` → 콘텐츠
- `./quartz.config.ts` → 사이트 설정
- `./quartz.layout.ts` → 레이아웃 설정

`quartz/` 폴더는 **마운트 안 됨**. 컨테이너 빌드 시점의 엔진 코드가 그대로 고정된다.

의도: 엔진은 안정적이라고 가정하고 콘텐츠 편집만 즉시 반영. GitHub Pages 배포 환경과 최대한 가깝게.

## dev 모드 (엔진 수정용)

`docker-compose.yml` + `docker-compose.dev.yml` 오버라이드.

dev 오버라이드가 `./quartz` 전체 디렉토리를 컨테이너의 `/usr/src/app/quartz`로 추가 마운트. 엔진 소스 수정도 즉시 반영된다.

필요한 상황:
- 디자인 테마 변경 (`quartz/styles/custom.scss`)
- 새 컴포넌트 추가 (`quartz/components/*`)
- 레이아웃 동작 수정
- Quartz 내부 플러그인 수정

## 왜 한 파일로 합치지 않았나

`./quartz`를 항상 마운트하는 단일 compose 파일로 끝내면 간단하다. 3가지 이유로 분리했다:

1. **배포 환경 재현성**: 기본 모드는 GitHub Pages 배포 빌드와 동일한 상태여야 한다. 엔진 볼륨 마운트는 미세한 차이를 만든다.
2. **실수 방지**: 엔진 파일을 무심코 고쳤을 때 즉시 반영되면 알아차리기 어렵다. 기본 모드에선 "엔진 수정 → 반영 안 됨"이라는 명확한 신호가 있다.
3. **Windows 성능**: Docker Desktop의 볼륨 마운트는 Windows에서 느릴 수 있다. 불필요한 마운트는 뺀다.

# 사전지식

- **Docker Compose 오버라이드**: `-f` 플래그로 compose 파일을 여러 개 지정하면 순서대로 병합된다. 뒤에 오는 파일이 앞 파일의 값을 덮어쓰거나 배열을 append한다. `volumes`는 append라서 기본 compose의 볼륨에 dev 볼륨이 더해지는 구조가 된다.
- **Claude Code Hook**: `.claude/settings.json`에 PreToolUse/PostToolUse 훅을 등록하면 Claude가 도구를 호출하기 전/후에 쉘스크립트가 실행된다. stdin으로 JSON payload를 받고, stderr 출력은 Claude에게 전달되며, exit 0은 진행 허용, 비영(非零)은 차단.
- **`docker inspect` 마운트 조회**: 컨테이너가 실제로 어떤 볼륨을 바인드했는지 `docker inspect <id> --format '{{range .Mounts}}...{{end}}'`로 확인 가능하다. CI/훅에서 상태 검증에 유용.

# 전체 코드

## docker-compose.yml (기본)

```yaml
services:
  quartz:
    build: .
    ports:
      - "8080:8080"
    volumes:
      - ./content:/usr/src/app/content
      - ./quartz.config.ts:/usr/src/app/quartz.config.ts
      - ./quartz.layout.ts:/usr/src/app/quartz.layout.ts
```

## docker-compose.dev.yml (오버라이드)

```yaml
services:
  quartz:
    volumes:
      - ./quartz:/usr/src/app/quartz
```

## .claude/hooks/check-engine-mount.sh

```bash
#!/bin/bash
# PreToolUse hook: quartz/ 내부 파일 편집 시 dev 모드 여부 검사.
# 꺼져있으면 stderr 경고. exit 0 — 블로킹 안 함.

set -e

payload=$(cat)
file=$(echo "$payload" | grep -o '"file_path"[[:space:]]*:[[:space:]]*"[^"]*"' | head -1 | sed 's/.*"file_path"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/')

[ -z "$file" ] && exit 0

case "$file" in
  */quartz/*|*\\quartz\\*) ;;
  *) exit 0 ;;
esac

command -v docker >/dev/null 2>&1 || exit 0

cid=$(docker ps --filter "label=com.docker.compose.service=quartz" --format '{{.ID}}' | head -1)
[ -z "$cid" ] && exit 0

mounts=$(docker inspect "$cid" --format '{{range .Mounts}}{{.Destination}}|{{.Type}}
{{end}}' 2>/dev/null || echo "")

if echo "$mounts" | grep -qE '^/usr/src/app/quartz\|'; then
  exit 0
fi

cat >&2 <<'EOF'
[check-engine-mount] quartz/ 내부 파일 편집 중인데 dev 모드가 아님.
  docker-compose down
  docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build
EOF
exit 0
```

## .claude/settings.json (훅 등록 부분)

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "bash .claude/hooks/check-engine-mount.sh" }
        ]
      }
    ]
  }
}
```

# 라인별 해설

## docker-compose.yml

- `ports: "8080:8080"`: 호스트와 컨테이너 포트를 통일. 이전엔 `18080:8080`이어서 Node 직접 실행(8080)과 숫자가 달라 혼동의 원인이었다. 통일 후 접속 URL은 항상 `http://localhost:8080`.
- `volumes`에 `./quartz`가 없는 이유: 엔진 고정. 빌드 시 `COPY . .`로 복사된 코드가 그대로 돌아간다.

## docker-compose.dev.yml

- `services.quartz.volumes`에 `./quartz:/usr/src/app/quartz` 한 줄만. 오버라이드 병합 규칙상 기본 compose의 볼륨 3개에 이 1개가 더해진다. 총 4개 마운트로 동작.
- dev 모드 시작 명령은 `docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build`. `-f` 순서가 병합 순서.

## check-engine-mount.sh

- `payload=$(cat)`: Claude가 hook에 JSON을 stdin으로 보낸다. 예: `{"tool_name":"Edit","tool_input":{"file_path":"quartz/components/ThemeSwitcher.tsx"}}`
- `grep/sed로 file_path 추출`: jq가 없는 환경(Windows Git Bash 기본 상태)을 고려해 grep+sed로 대체. 정규식이 투박해 보이지만 의존성 제로.
- `case "$file" in */quartz/* ...`: quartz 폴더 내부 파일만 검사. `quartz.config.ts`·`quartz.layout.ts`는 루트 파일이라 이 패턴에 안 걸리므로 제외된다.
- `docker ps --filter "label=com.docker.compose.service=quartz"`: docker-compose가 붙인 레이블로 현재 실행 중인 quartz 서비스 컨테이너 ID를 가져온다. 컨테이너 이름 하드코딩 대신 이 방식을 쓰면 프로젝트 이름 바뀌어도 동작.
- `docker inspect --format '{{range .Mounts}}{{.Destination}}|{{.Type}}...'`: 마운트 목록을 `Destination|Type` 형식으로 줄바꿈 나열.
- `grep -qE '^/usr/src/app/quartz\|'`: 정확히 `/usr/src/app/quartz` 디렉토리 마운트가 있는지 확인. `quartz.config.ts` 같은 파일 마운트는 `/usr/src/app/quartz.config.ts|bind` 형태라 패턴에 안 걸린다.
- `exit 0`으로 마무리: 훅은 경고만 띄우고 블로킹하지 않는다. Claude가 경고 보고 dev 모드로 전환한 뒤 재시도한다.

## settings.json

- `PreToolUse`에 `Write|Edit` matcher로 등록. 모든 파일 편집 직전에 발동.
- 기존 `PostToolUse`(validate-md.sh)와 독립. 두 훅이 서로 간섭하지 않는다.

# AI 자동화 원리

Claude Code는 대화 맥락을 읽어 모드를 **자동 판단**한다. 사용자가 매번 "dev 모드로 바꿔줘"라고 말할 필요 없다.

## 판단 규칙 (CLAUDE.md `## 로컬 실행` 섹션에 기재)

dev 모드 전환 신호:
- "디자인", "테마", "색상", "레이아웃", "폰트"
- "컴포넌트 추가/수정"
- "`quartz/` 안의 파일 수정"
- "스타일 변경"

기본 모드 복귀 신호:
- "디자인 확정", "이제 콘텐츠 작업"
- "커밋할게", "배포해"
- 대화 주제가 엔진 → 콘텐츠로 명확히 바뀔 때

## 안전망 — 훅

Claude가 맥락을 놓쳐도 훅이 `quartz/**` 편집 순간에 dev 모드 여부를 검사한다. 꺼져있으면 stderr 경고 → Claude가 읽고 dev 모드로 전환 후 재시도.

## 판단 규칙과 훅 둘 다 필요한 이유

훅은 **편집 순간**에만 발동한다. 편집 *전*에 대화 맥락만 봐도 "엔진 작업이구나" 판단 가능하면 미리 dev 모드로 올려두는 게 효율적이다. 훅은 판단 실패 시의 안전망 역할. 둘 다 있어야 "사용자가 직접 요청할 일 없음"이 성립한다.

# 교훈

- **프로젝트에 Dockerfile·compose 파일이 있으면 먼저 그것을 사용한다.** 추측으로 Node 직접 실행하지 않는다. 이번에 이 실수를 해서 사용자가 "로컬에 안 뜬다"를 반복 겪게 만들었다.
- **개발 편의를 위해 배포 환경 볼륨 마운트를 영구 추가하지 않는다.** 배포 환경과의 차이를 숨기면 나중에 "배포에선 왜 달라?" 디버깅이 어려워진다. 대신 모드를 분리한다.
- **자동화는 AI 판단 + 훅 안전망의 조합이다.** 한쪽만으로는 사용자 편의와 안정성을 동시에 달성하기 어렵다.
- **크로스-PC로 유지돼야 하는 규칙은 memory가 아니라 git 추적 파일에 둔다.** memory는 이 PC에서만 읽힌다. 집 PC에서 `git pull` 후에도 규칙이 적용되려면 CLAUDE.md·docs/·content/에 있어야 한다.
- **CLAUDE.md는 규칙만, 배경은 content/에.** CLAUDE.md는 매 세션 자동 로드되므로 토큰 비용이 든다. "왜 이렇게 했나" 같은 설명은 학습 노트로 분리해서 필요할 때만 읽히게 한다.
