# git 기본 명령어

## 개요
> 브랜치 분기·머지·강제 덮어쓰기 등 실전 작업에서 자주 쓰는 기본 명령어 흐름 정리.

## 1. 새 브랜치 분기 및 원격 연결

`dev` 기준으로 작업용 브랜치를 새로 만들어 원격까지 연결하는 전형적인 흐름이다.

```bash
# 1. 기준 브랜치로 이동
git checkout dev

# 2. 원격 dev 최신 상태 반영
git pull

# 3. dev 기준으로 새 브랜치 생성 및 전환
git checkout -b jintae-branch

# 4. 원격에 새 브랜치 업로드 + tracking 설정
git push -u origin jintae-branch
```

- `git checkout -b` — 현재 브랜치 상태를 복사해 새 브랜치 생성 후 전환.
- `git push -u` — 로컬 브랜치와 원격 브랜치를 tracking 관계로 연결. 이후부터는 `git push`만 쳐도 자동으로 같은 원격 브랜치로 간다.

## 2. 작업 브랜치를 dev에 머지

`jintae-branch`에서 작업이 끝난 뒤 `dev`로 반영하는 표준 흐름이다.

```bash
# 1. 대상 브랜치로 이동
git checkout dev

# 2. 원격 최신 상태 반영
git pull

# 3. 작업 브랜치 머지
git merge jintae-branch

# 4. 충돌 발생 시 해결 후 커밋
git add .
git commit

# 5. 원격 반영
git push
```

- 머지 전 반드시 `git pull`로 `dev`를 최신화한다. 그러지 않으면 불필요한 충돌이 발생한다.
- 충돌이 없으면 `git merge` 단계에서 자동으로 머지 커밋이 생성된다.

## 3. 작업 브랜치 내용으로 dev 강제 덮어쓰기

> [!WARNING]
> `git reset --hard` + `git push -f`는 기존 이력을 **완전히 파괴**한다. 협업 브랜치(`main`, `dev`)에서는 다른 사람이 올린 커밋도 함께 소실된다. **혼자 쓰는 브랜치거나 팀 전원의 합의가 있을 때만** 사용한다.

`dev` 이력을 완전히 버리고 `jintae-branch` 내용으로 교체하는 경우이다.

```bash
# 1. 대상 브랜치로 이동
git checkout dev

# 2. jintae-branch 상태로 강제 리셋
git reset --hard jintae-branch

# 3. 원격 dev에도 강제 반영
git push -f origin dev
```

- `reset --hard`는 작업 트리까지 함께 되돌리므로 커밋 이력과 로컬 변경사항이 모두 사라진다.
- `push -f`는 원격 이력을 덮어쓴다. 다른 사람이 이미 그 브랜치에서 작업 중이라면 그 작업까지 파괴된다.
