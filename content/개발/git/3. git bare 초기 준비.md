# Git Bare 초기 준비

본 문서는 Git Bare 저장소를 세팅하기 **전에** 반드시 선행되어야 하는 설치·인증·접속 준비 과정을 정리한 가이드이다. 모든 명령어는 **Git Bash** 기준으로 작성되었다.

> **💡 구동 환경 안내 (Git Bash 쉘)**
> 본 가이드의 모든 터미널 명령어는 윈도우용 **Git Bash 쉘(Shell)**을 기준으로 작성되었다. Git Bash는 Windows 위에서 동작하지만 **리눅스(Linux) 기반의 명령어 생태계**(`ls`, `mkdir`, `/` 형태의 경로 구조 등)를 그대로 사용한다.

---

## [STEP 1] 개인 PC: Git 설치 확인

가장 먼저, 내 PC에 Git이 깔려있는지 확인한다.

```bash
# [개인 PC - Git Bash]
git --version
```

> **잠깐, 여기서 `git`은 뭔가?** 터미널에 `git`이라고 치면, 쉘이 PC 내부의 `PATH`(실행파일 검색 경로 목록)를 쭉 훑어서 `git.exe`라는 **실행파일(프로그램)**을 찾아 실행시킨다. 즉 `git --version`은 **"Git아, 너 버전이 뭐야?"** 라고 묻는 것이다.
>
> ```bash
> # 실제로 git이 어디 깔려있는지 보고 싶으면:
> type -a git    # → git is /mingw64/bin/git
> which git      # → /mingw64/bin/git
> ```
>
> 이 Git 프로그램은 GitHub, GitLab, 사내 서버 등 **어디에서든 공통으로 사용하는 하나의 코어 엔진**이다. "GitHub 전용 Git", "GitLab 전용 Git" 같은 것은 존재하지 않는다. GitHub도, 우리가 지금 만들려는 Bare 서버도, 전부 이 같은 Git 프로그램이 뒤에서 돌아가는 것이다.

---

## [STEP 2] 개인 PC: SSH 키 생성

Git Bare 서버에 코드를 push할 때마다 비밀번호를 치는 건 고문이다. SSH 공개키 인증을 세팅해두면 앞으로 **비밀번호 입력 없이** 서버에 접속하고 push/pull 할 수 있게 된다.

```bash
# [개인 PC - Git Bash]
ssh-keygen -t ed25519
```

> **`ssh-keygen`은 뭔가?** 이건 Git과는 **전혀 별개인 프로그램**이다. **OpenSSH**라는 소프트웨어에 포함된 실행파일로, SSH 키를 생성해주는 역할을 한다.
>
> 아무 컴퓨터에나 항상 깔려있는 것은 아니지만, Git for Windows를 설치하면 OpenSSH 클라이언트가 같이 딸려와서 Git Bash에서는 대부분 바로 쓸 수 있다.
>
> 혹시 없을 수도 있으니 아래로 확인한다:
>
> ```bash
> which ssh-keygen   # → /usr/bin/ssh-keygen 같은 경로가 나오면 정상
> ssh -V             # OpenSSH 버전 확인
> ```
>
> 경로가 안 나오면 Git for Windows를 재설치하거나, Windows 설정 > 앱 > 선택적 기능에서 "OpenSSH 클라이언트"를 추가 설치하면 된다.

> **`ssh-keygen -t ed25519` 해부:**
>
> ```
> ssh-keygen   -t      ed25519
> ─────────    ──      ───────
>    ①          ②        ③
> ```
>
> | 번호 | 역할 | 설명 |
> |---|---|---|
> | ① `ssh-keygen` | **실행파일 이름** | SSH 키를 생성해주는 프로그램 (OpenSSH 소속) |
> | ② `-t` | **옵션** | "type" — 어떤 알고리즘으로 키를 만들지 지정 |
> | ③ `ed25519` | **옵션값** | 구체적인 알고리즘 이름. RSA보다 키가 짧으면서도 보안이 강력하고, 생성/검증 속도도 빠른 현대적 추천 알고리즘 |

- 첫 번째 질문 (저장 경로): **그냥 Enter**. 기본 경로 `~/.ssh/id_ed25519`에 저장된다.
- 두 번째 질문 (passphrase): **그냥 Enter**. 편의 우선이면 비워두면 된다. (보안을 더 강화하고 싶으면 암호 입력)

이렇게 하면 **두 개의 파일**이 생긴다:

| 파일 | 위치 | 역할 |
|---|---|---|
| `id_ed25519` | `~/.ssh/id_ed25519` | **비밀키** |
| `id_ed25519.pub` | `~/.ssh/id_ed25519.pub` | **공개키** |

> **`~`는 무슨 경로?** 현재 로그인한 사용자의 홈 디렉토리를 뜻한다. Windows Git Bash에서는 보통 `C:\Users\내계정명`이다.

```bash
# [개인 PC - Git Bash]
# 공개키 내용 확인 & 복사
cat ~/.ssh/id_ed25519.pub
# → ssh-ed25519 AAAAC3Nza... 어쩌구 (이 한 줄 전체를 드래그해서 복사!)
```

## [STEP 3] 서버 PC: 공개키 등록

이제 내 개인 PC의 Git Bash에서 서버로 접속한 다음, 서버 안에 공개키를 등록해준다.

```bash
# [개인 PC - Git Bash] 여기서 서버로 SSH 접속!
ssh gitid@10.10.10.10
# (이 시점부터는 서버 셸 안에 들어와 있는 상태)
```

```bash
# [서버 셸 - SSH 접속 후] 서버 안에서 아래를 실행

# 1. SSH 전용 디렉토리 생성 (서버당 최초 1번만 하면 됨)
mkdir -p ~/.ssh
chmod 700 ~/.ssh
```

> **왜 하필 `~/.ssh` 인가?** SSH 프로그램이 키를 찾을 때, **무조건 해당 사용자 홈 아래 `.ssh` 폴더**를 들여다보게 하드코딩 되어있다(정확히는 SSH 설정에서 바꿀 수 있지만 99%는 기본값 사용). 아무 폴더에 키를 넣어봤자 SSH가 찾지를 못한다. 그래서 이 경로를 쓰는 것이다.
>
> 여기서 `~`는 루트(`/`)가 아니다. **현재 로그인한 사용자 홈 디렉토리**를 뜻한다.
>
> `chmod 700`은 **"이 폴더는 주인만 읽고 쓸 수 있게 잠가!"**라는 뜻이다. SSH는 보안에 예민해서, 이 권한이 안 맞으면 키 인증 자체를 거부하기도 한다.

```bash
# [서버 셸]

# 2. 공개키 등록 파일 열기
vi ~/.ssh/authorized_keys
```

- `i`를 눌러 입력 모드 진입 → 아까 복사한 공개키 한 줄 붙여넣기 → `Esc` → `:wq` → Enter로 저장.
- 이미 다른 키가 적혀있다면, **새 줄에 추가**하면 된다. (한 줄에 하나의 키)

```bash
# [서버 셸]

# 3. 권한 설정 (가능하면 적용)
chmod 600 ~/.ssh/authorized_keys
# "이 파일도 주인만 읽고 쓸 수 있게 잠가!"
```

## [STEP 4] 개인 PC: 비밀번호 없이 접속 테스트

서버 셸에서 `exit`로 빠져나온 뒤, 다시 접속해본다.

```bash
# [개인 PC - Git Bash]
ssh gitid@10.10.10.10
```

- 비밀번호를 안 물어보고 바로 접속되면 **성공!**
- 여전히 비밀번호를 묻는다면: 서버의 `~/.ssh` 폴더 및 `authorized_keys` 파일 권한(700, 600)을 다시 확인한다.

---

## [STEP 5] 서버: Git 설치 확인

```bash
# [서버 셸]
git --version
```

- 버전이 출력되면 이미 설치된 상태. 넘어가면 된다.
- `command not found`가 뜨면 서버에 Git이 없는 것이므로, [git-scm.com](https://git-scm.com/) 에서 다운로드하여 설치한다.

## [STEP 6] 서버: Bare 저장소 생성

```bash
# [서버 셸]

# 1. 저장소들을 모아둘 폴더 + 프로젝트 폴더 생성
mkdir -p ~/repos/jintae.git
cd ~/repos/jintae.git

# 2. Bare 저장소로 초기화
git init --bare
```

> **`git init --bare` vs 그냥 `git init` — 저장소 생성 모드 2가지:**
>
> Git에서 저장소를 만드는 방법은 딱 **2가지 모드**뿐이다.
>
> | 명령어 | 이름 | 내부 구조 | 용도 |
> |---|---|---|---|
> | `git init` | **일반(Non-Bare) 저장소** | 소스 파일이 눈에 보이는 워킹 디렉토리 + 숨겨진 `.git` 폴더 | 개발자가 직접 코드를 편집하고 커밋하는 **작업용** |
> | `git init --bare` | **Bare 저장소** | 워킹 디렉토리 없이 `.git` 내부 구조만 노출 | 코드 보관·공유 전용 **중앙 허브**. 직접 편집 불가 |
>
> 우리가 로컬 PC에서 `git init`으로 만드는 건 일반 저장소이다. 파일을 직접 열어보고 수정하고 커밋할 수 있다.
>
> 반면 서버에 만드는 Bare 저장소는 소스 파일이 풀려있지 않다. 오직 push/pull을 받기 위한 "코드 보관 전용 창고" 역할만 한다. **GitHub 내부 서버도 이 Bare 방식으로 저장소를 운영**한다. 우리가 하는 건 GitHub가 해주던 걸 직접 꾸리는 것이다.

- **`~/repos`는 Git이 자동으로 만들어주는 폴더가 아니다.** 여러 저장소를 한 곳에 깔끔히 모아두기 위해 **사용자가 직접 만든 정리용 폴더**이다. 이름도 `repos`가 아니라 `my-gits`든 `storage`든 뭐든 상관없다.
- **`.git` 접미사 관례**: Bare 저장소 폴더 이름 끝에 `.git`을 붙이는 것은 "이 폴더는 Bare 저장소다"라고 표시해주는 **업계 관례**일 뿐, 필수는 아니다.
