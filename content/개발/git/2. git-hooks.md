# Git Hooks 완전 가이드

> bcmf 프로젝트의 자동 배포 훅 설정 경험을 바탕으로 작성된 문서입니다.  
> 원리 이해, 스크립트 기반 배포 흐름 파악, 그리고 과거의 시행착오를 통한 학습을 목적으로 재구성되었습니다.

---

## 1. Git 훅의 기본 개념

Git 훅(Hook)은 **특정 Git 이벤트가 발생할 때 자동으로 실행되는 스크립트**입니다.

예를 들어 `git push`를 받았을 때, `git commit`을 할 때, `git merge` 전후에 원하는 작업을 자동으로 수행할 수 있습니다.

```
개발자 로컬                        서버 bare repo
─────────────────────────────────────────────────────
git push origin main  ──────────►  코드 수신
                                       ↓
                                  post-receive 훅 자동 실행
                                       ↓
                                  빌드 + PM2 재시작
```

### 훅 파일 위치

훅 파일은 **`.git/hooks/`** 폴더(로컬) 또는 **`bare_repo/hooks/`** 폴더(서버)에 위치합니다.

```
repos/atlas-demo.git/
└── hooks/
    ├── applypatch-msg.sample
    ├── commit-msg.sample
    ├── post-receive          ← 우리가 만든 파일 (확장자 없음!)
    ├── pre-commit.sample
    ├── pre-push.sample
    └── ...
```

---

### 훅의 두 가지 종류

| 종류 | 실행 위치 | 대표 훅 |
|---|---|---|
| **클라이언트 훅** | 개발자 로컬 `.git/hooks/` | `pre-commit`, `commit-msg`, `pre-push` |
| **서버 훅** | bare repo의 `hooks/` | `pre-receive`, `post-receive`, `update` |

---

### 주요 훅 파일 설명

#### 서버 훅 (배포/자동화에 사용)

**1. `pre-receive`**
- **실행 시점**: push를 받기 직전
- **용도**: push 자체를 허용/거부하는 게이트키퍼 역할
- **종료 코드**: `exit 1`로 종료하면 push가 거부됨
- **활용 예**: 커밋 메시지 컨벤션 강제, 브랜치 보호, 파일 크기 제한

```bash
#!/bin/bash
# 예: main 브랜치에 직접 push 금지
while read oldrev newrev ref; do
  if [ "$ref" = "refs/heads/main" ]; then
    echo "ERROR: main 브랜치에 직접 push할 수 없습니다. PR을 사용하세요."
    exit 1
  fi
done
```

**2. `update`**
- **실행 시점**: 각 브랜치가 업데이트될 때 (브랜치마다 1회)
- **용도**: 특정 브랜치별 동작 제어
- **차이점**: `pre-receive`는 push 전체에 1회, `update`는 브랜치마다 실행

**3. `post-receive` ⭐ (우리가 사용한 배포 훅)**
- **실행 시점**: push가 완전히 완료된 후
- **용도**: 배포 자동화, 알림 발송, CI 트리거
- **특징**: 이미 push가 완료된 후라 거부 불가. 부작용 없는 처리에 적합
- **stdin 입력**: `이전커밋해시 새커밋해시 refs/heads/브랜치명` 형식으로 받음

> **checkout이란?**  
> `git checkout main`은 **"main 브랜치 기준으로 파일들을 실제 워킹트리에 꺼내 놓아라"** 는 뜻입니다.  
> Git 내부에 압축 저장된 파일 오브젝트들을 실제 폴더의 파일로 변환해서 배치하는 행위입니다.

#### 클라이언트 훅 (코드 품질 관리에 사용)

**1. `pre-commit`**
- **실행 시점**: `git commit` 명령 실행 직후, 커밋 메시지 입력 전
- **용도**: 코드 린트, 포맷 검사, 테스트 실행
- **exit 1**: 커밋 자체를 중단시킴

```bash
#!/bin/bash
# 예: ESLint 통과 못하면 커밋 차단
npm run lint
if [ $? -ne 0 ]; then
  echo "ESLint 오류가 있습니다. 커밋을 중단합니다."
  exit 1
fi
```

**2. `commit-msg`**
- **실행 시점**: 커밋 메시지가 작성된 후
- **용도**: 커밋 메시지 컨벤션 강제 (예: Conventional Commits)

**3. `pre-push`**
- **실행 시점**: `git push` 실행 직후, 실제 전송 전
- **용도**: push 전 최종 테스트 실행

## 2. bcmf 전체 배포 흐름도

Git 훅이 언제 동작해서 어떤 흐름을 만들어내는지 아키텍처를 그리면 다음과 같습니다.

```
[로컬]                          [서버]
  │                               │
  │  git push origin main         │
  ├──────────────────────────────►│
  │                               │
  │                         bare repo 수신
  │                               │
  │                         post-receive 실행
  │                               │
  │                         main 브랜치 확인
  │                               │
  │                    ┌──────────┴──────────┐
  │                    │  변경 파일 분석      │
  │                    │  (oldrev → newrev)   │
  │                    └──────────┬──────────┘
  │                               │
  │              ┌────────────────┼────────────────┐
  │              │                │                │
  │        backend/ 변경?   frontend/ 변경?   둘 다 없음?
  │              │                │                │
  │         빌드+PM2재시작    빌드만 수행        스킵
  │              │                │                │
  │              └────────────────┼────────────────┘
  │                               │
  │◄──────────────────────────────┤
  │   배포 완료 로그 출력          │
```

## 3. 코드를 읽기 위한 사전 지식: Bash 문법

본격적인 훅 코드를 해석하기 전에, 스크립트 작성에 쓰인 핵심 문법을 익혀야 합니다.

```bash
# ── 변수 선언과 사용 ──────────────────────────────
WORK_TREE="/d/srv/atlas-demo"    # 선언: = 양쪽 공백 없어야 함!
echo $WORK_TREE                   # 사용: $ 붙이면 값이 꺼내짐 → 화면에 출력
echo "$WORK_TREE/backend"         # 따옴표: 경로에 공백 있어도 안전하게 처리

# ── 명령어 결과를 변수에 저장 ────────────────────
BRANCH=$(echo $ref | cut -d/ -f3)
# $() 안의 명령을 실행하고, 그 출력을 BRANCH에 저장
# echo  = 말하기 (화면에 출력)
# $()   = 녹음기 (말한 걸 받아서 변수에 저장)

# ── if 조건문 ──────────────────────────────────────
if [ "$BRANCH" != "main" ]; then
  echo "스킵"
fi
# [ ] 는 조건 검사
# !=  는 "같지 않다"
# then ~ fi 가 if 블록의 시작과 끝

# ── while read 루프 ────────────────────────────────
while read oldrev newrev ref; do
  # 한 줄씩 읽어서 oldrev, newrev, ref 세 변수에 나눠 저장
  # 입력이 들어오는 동안 반복 실행
done    # 루프 끝

# ── && 단축 평가 ────────────────────────────────────
grep -q "^backend/" && BACKEND_CHANGED=true
# 왼쪽 성공(=파일 발견) → 오른쪽 실행
# 왼쪽 실패(=파일 없음) → 오른쪽 건너뜀

# ── 변수로 명령 실행 ────────────────────────────────
PM2="/c/Users/.../pm2"
$PM2 restart node/atlas-demo    # $PM2 자리에 경로가 들어가서 실제 pm2 실행

# ── | 파이프 ────────────────────────────────────────
echo $ref | cut -d/ -f3
# 왼쪽 출력을 오른쪽 명령어의 입력으로 연결
# echo 가 "refs/heads/main" 을 내보내면, cut 이 받아서 처리
```

## 4. bcmf 배포 훅 적용 및 전체 코드

배포 훅이 어떻게 적용되어 있는지, 전체 코드는 어떤 완성본을 가지고 있는지 확인합니다.

### 배포 환경 정보 요약
훅 스크립트에 포함되는 설정 정보들은 서버 내 아래 경로에 위치하고 있습니다.
| 항목 | 내용 |
|---|---|
| bare repo 위치 (`GIT_DIR`) | `C:\Users\deploysvc\repos\atlas-demo.git` |
| 워킹 디렉토리 (`WORK_TREE`) | `D:\srv\atlas-demo` |
| 훅 파일 위치 | `C:\Users\deploysvc\repos\atlas-demo.git\hooks\post-receive` |
| 배포 대상 브랜치 | `main` |
| PM2 앱 이름 | `node/atlas-demo` |
| 서버 환경 | Windows + Git Bash |

### 훅 활성화를 위한 필수 설정
> [!IMPORTANT]
> **훅 활성화를 위한 필수 단계**
> 1. **확장자 제거**: `.sample` 확장자가 붙은 파일은 비활성화 상태입니다. 확장자를 제거해야 Git이 인식합니다.  
>    (예: `post-receive.sample` → `post-receive`)
> 2. **실행 권한 부여**: 파일에 **실행 권한(`chmod +x`)**이 있어야 스크립트가 돌아갑니다.

#### 💡 실행 권한 부여 예시 (Git Bash)
```bash
chmod +x /c/Users/deploysvc/repos/atlas-demo.git/hooks/post-receive
```

### 전체 코드 (`post-receive`)
```bash
#!/bin/bash

# ========================================
# bcmf post-receive 배포 훅 (선택적 빌드)
# ========================================

WORK_TREE="/d/srv/atlas-demo"
GIT_DIR="/c/Users/deploysvc/repos/atlas-demo.git"
PM2="/c/Users/deploysvc/AppData/Roaming/npm/pm2"

while read oldrev newrev ref; do
  BRANCH=$(echo $ref | cut -d/ -f3)

  if [ "$BRANCH" != "main" ]; then
    echo ">>> [SKIP] main 브랜치가 아님 ($BRANCH) — 배포 건너뜀"
    exit 0
  fi

  echo ""
  echo "================================================"
  echo ">>> [bcmf 배포 시작] $(date '+%Y-%m-%d %H:%M:%S')"
  echo "================================================"

  echo ">>> [1/4] 코드 반영 중..."
  git --work-tree="$WORK_TREE" --git-dir="$GIT_DIR" checkout -f main

  CHANGED=$(git --git-dir="$GIT_DIR" diff --name-only $oldrev $newrev)
  echo ">>> 변경된 파일:"
  echo "$CHANGED" | sed 's/^/    /'

  BACKEND_CHANGED=false
  FRONTEND_CHANGED=false

  echo "$CHANGED" | grep -q "^backend/"  && BACKEND_CHANGED=true
  echo "$CHANGED" | grep -q "^frontend/" && FRONTEND_CHANGED=true

  if [ "$BACKEND_CHANGED" = true ]; then
    echo ">>> [2/4] 백엔드 변경 감지 → 빌드 시작"
    cd "$WORK_TREE/backend"
    npm install --omit=dev
    npm run build
    NEED_PM2_RESTART=true
  else
    echo ">>> [2/4] 백엔드 변경 없음 — 빌드 스킵"
    NEED_PM2_RESTART=false
  fi

  if [ "$FRONTEND_CHANGED" = true ]; then
    echo ">>> [3/4] 프론트엔드 변경 감지 → 빌드 시작"
    cd "$WORK_TREE/frontend"
    npm install --omit=dev
    npm run build
  else
    echo ">>> [3/4] 프론트엔드 변경 없음 — 빌드 스킵"
  fi

  if [ "$NEED_PM2_RESTART" = true ]; then
    echo ">>> [4/4] PM2 재시작 중..."
    $PM2 restart node/atlas-demo
  else
    echo ">>> [4/4] PM2 재시작 불필요 (백엔드 미변경)"
  fi

  echo ">>> [배포 완료] $(date '+%Y-%m-%d %H:%M:%S')"
done
```

## 5. 심층 분석: 코드 라인별 동작 해설

위에서 살펴본 코드가 실제로 어떻게 동작하는지 라인별로 정밀하게 분석해 봅니다.

### 환경 변수 설정
```bash
WORK_TREE="/d/srv/atlas-demo"
GIT_DIR="/c/Users/deploysvc/repos/atlas-demo.git"
PM2="/c/Users/deploysvc/AppData/Roaming/npm/pm2"
```

| 변수 | 설명 |
|---|---|
| `WORK_TREE` | 실제 소스코드가 있는 워킹 디렉토리 (Windows `D:\srv\atlas-demo` → Git Bash 경로) |
| `GIT_DIR` | bare repo 경로 (Windows `C:\Users\deploysvc\repos\atlas-demo.git`) |
| `PM2` | pm2 실행 파일 전체 경로. 훅은 일반 쉘과 달리 PATH가 제한적이라 전체 경로 필요 |

> [!TIP]
> **PM2 실제 경로 확인하기**  
> 서버(Git Bash)에서 `which pm2` 명령어를 실행하면 실제 배포 서버의 PM2 위치가 출력됩니다. 이 경로를 위 `PM2="..."` 변수 선언부에 절대 경로로 기입해 주세요.

> **왜 `WORK_TREE`라고 부를까요? (Working Tree)**  
> 파일 시스템의 구조가 나무(트리) 모양이기 때문에 Git 공식 용어로 "Working Tree(작업 트리)"라고 부릅니다.  
> 의미상으로는 "워킹 디렉토리"와 같습니다.
> ```
> D:\srv\atlas-demo\      <- 루트 (줄기)
> +-- backend\            <- 가지
> |   +-- src\
> |       +-- index.ts   <- 잎 (leaf)
> +-- frontend\
> ```

> **왜 Git Bash 경로로 쓰나요?**  
> 훅은 Git Bash 환경에서 실행됩니다. Git Bash는 Windows 경로(`C:\`)를 Unix 스타일(`/c/`)로 변환해서 사용합니다.

---

### stdin 루프
```bash
while read oldrev newrev ref; do
```
`post-receive`는 push된 브랜치 수만큼 stdin으로 입력을 받습니다.  
예: `main`과 `develop` 브랜치를 동시에 push하면 2줄이 들어옵니다.

```
# 이전해시(oldrev)     새로운해시(newrev)     브랜치명(ref)
a1b2c3d4e5...      f6g7h8i9j0...      refs/heads/main
x1y2z3w4v5...      p6q7r8s9t0...      refs/heads/develop
```

| 변수 | 내용 | 데이터 예시 (main 1줄 기준) |
|---|---|---|
| `oldrev` | push하기 **이전** 마지막 커밋 해시 | `a1b2c3d4e5...` (실제로는 40자) |
| `newrev` | 이번에 새로 들어온 **최신** 커밋 해시 | `f6g7h8i9j0...` (실제로는 40자) |
| `ref` | 업데이트된 브랜치 전체 경로 | `refs/heads/main` |

> **커밋 해시(Commit Hash)란?**  
> Git은 커밋(코드 저장) 단위마다 **40자리 고유 영문/숫자 ID**를 자동으로 생성합니다. 지문 같은 것으로, 해시가 같으면 100% 같은 내용의 커밋입니다.  
> (설명의 편의상 위 예시에서는 `a1b2c3d4e5...` 처럼 줄여서 표현했습니다.)
> ```
> commit d3f8a2b1c4e9f0a7b3c5d8e2f4a1b6c9d0e3f7a2  <- 실제 커밋 해시 예시 (40자)
> Author: 홍길동
> Message: feat: 로그인 기능 추가
> ```
> `oldrev -> newrev` 이 두 개의 해시가 있으면 "과거 커밋부터 최신 커밋까지 어떤 파일이 바뀌었는지"를 계산할 수 있습니다.

---

### 브랜치 필터링
```bash
BRANCH=$(echo $ref | cut -d/ -f3)
if [ "$BRANCH" != "main" ]; then
  exit 0
fi
```

`refs/heads/main`에서 `/`를 구분자로 3번째 필드(`main`)만 추출합니다.  
`main`이 아닌 브랜치(예: `develop`, `feature/xxx`) push 시에는 배포 없이 종료합니다.

---

### 코드 반영 (핵심)
```bash
git --work-tree="$WORK_TREE" --git-dir="$GIT_DIR" checkout -f main
```

`git checkout main`은 **"main 브랜치 기준으로 파일들을 실제 워킹트리에 꺼내 놓아라"** 는 뜻입니다.

| 명령어 | 설명 |
|---|---|
| `--work-tree` | bare repo의 최신 코드를 풀어서 **실제로 배치할 도착지(목적지) 폴더**를 지정합니다. (`D:\srv\atlas-demo` 폴더 안에 최신 코드 파일들이 그대로 깔립니다.) |
| `--git-dir` | git 메타데이터(커밋, 브랜치 정보 등)가 있는 곳 지정 (bare repo) |
| `checkout -f` | 강제로 최신 상태로 덮어씀 (`-f` = force, 로컬 변경 무시) |

---

### 변경 파일 감지
```bash
CHANGED=$(git --git-dir="$GIT_DIR" diff --name-only $oldrev $newrev)
echo "$CHANGED" | grep -q "^backend/"  && BACKEND_CHANGED=true
echo "$CHANGED" | grep -q "^frontend/" && FRONTEND_CHANGED=true
```

| 코드 | 설명 |
|---|---|
| `diff --name-only $oldrev $newrev` | 이전 커밋과 새 커밋 사이에서 변경된 파일 이름만 추출 |
| `grep -q "^backend/"` | `backend/`로 시작하는 파일이 있는지 조용히(`-q`) 검색 |
| `&& BACKEND_CHANGED=true` | grep이 성공(0 반환)하면 변수를 true로 설정 |

출력 예시:
```
backend/src/routes/user.ts
frontend/src/pages/Login.tsx
frontend/src/components/Header.tsx
```

---

**`&&` 단축 평가 — 정확한 이해**

`&&`는 "왼쪽이 성공하면 오른쪽을 실행해라" 는 뜻입니다.

```bash
# 구조
왼쪽_명령어 && 오른쪽_명령어

# 왼쪽 성공(=파일 발견) → 오른쪽 실행
# 왼쪽 실패(=파일 없음) → 오른쪽 건너뜀
```

풀어서 읽으면:
```bash
echo "$CHANGED" | grep -q "^backend/" && BACKEND_CHANGED=true
# → "변경 파일 중 backend/로 시작하는 게 있으면 → BACKEND_CHANGED를 true로 바꿔라"
```

---

### 선택적 빌드 로직
```bash
if [ "$BACKEND_CHANGED" = true ]; then
  cd "$WORK_TREE/backend"
  npm install --omit=dev   # devDependencies 제외하고 설치 (프로덕션)
  npm run build            # TypeScript → JavaScript 컴파일
  NEED_PM2_RESTART=true
fi
```

| 명령어 | 설명 |
|---|---|
| `npm install --omit=dev` | `devDependencies`(테스트, 타입, 빌드 도구 등)를 제외하고 설치. 프로덕션 서버에서 불필요한 패키지 최소화 |
| `npm run build` | `backend/tsconfig.json` 기준으로 TypeScript를 `backend/dist/`에 컴파일 |

> **프론트엔드는 PM2 재시작이 불필요한 이유**  
> `frontend/dist/`에 정적 파일(HTML, JS, CSS)이 생성되면, Express 백엔드가 이미 해당 폴더를 서빙하고 있습니다.  
> 파일만 갱신되면 다음 요청부터 자동으로 새 파일을 사용하므로 재시작 불필요합니다.

---

### PM2 재시작
```bash
if [ "$NEED_PM2_RESTART" = true ]; then
  $PM2 restart node/atlas-demo
fi
```

`node/atlas-demo`는 `ecosystem.production.config.cjs`에서 정의된 앱 이름입니다.  
PM2가 현재 실행 중인 프로세스를 새 빌드 파일(`backend/dist/index.js`)로 재시작합니다.

> **⚠️ `node/atlas-demo` 이름을 바꾸면?**  
> 이 이름은 ecosystem 파일과 훅 스크립트가 **같이 공유**합니다. 이름을 변경하려면 **두 파일 모두 동시에** 수정해야 합니다.
> ```
> ecosystem.production.config.cjs  ->  name: "node/atlas-demo"  <- 여기서 정의
> post-receive (훅)                ->  $PM2 restart node/atlas-demo  <- 여기서 참조
> ```
> ecosystem만 바꾸고 훅을 안 바꾸면 `pm2 restart` 실패, 반대도 마찬가지입니다.

## 6. 경험과 교훈: 트러블슈팅 (실패 원인 분석)

초기 설정 시 겪었던 근본적인 오류들과, 이를 통해 배운 Git 작동 원리입니다.

### ❌ 실패 1: 훅 내부에서 `git pull` 사용

```bash
# 시도했던 방법 (예상과 다르게 동작함)
cd /d/srv/atlas-demo
git pull origin main
```

**구조 이해부터:**

```
[서버 내부 구조]

C:\Users\deploysvc\repos\atlas-demo.git   ← bare repo (git 저장소 본체)
         │
         │ git clone 으로 복사됨
         ▼
D:\srv\atlas-demo                 ← 워킹 디렉토리 (실제 코드 파일 있음)
   ├── backend/
   ├── frontend/
   └── .git/  ← 이 안에 origin = bare repo 주소 저장됨
```

`D:\srv\atlas-demo`는 clone된 정상 repo이므로 `git pull` 자체는 원래 가능합니다.  
**그러나** 훅이 실행될 때 Git이 환경변수를 자동으로 설정합니다:

```
[훅 실행 순간 일어나는 일]

  git push (로컬)
       │
       ▼
  atlas-demo.git 수신
       │
       ▼
  post-receive 훅 실행
  ┌─────────────────────────────────────────────────────┐
  │ Git이 자동으로 환경변수를 설정                       │
  │                                                     │
  │  GIT_DIR = "/c/Users/deploysvc/repos/atlas-demo.git"  ◄─ 여기! │
  │                                                     │
  │  이 상태에서 git pull을 실행하면...                  │
  │                                                     │
  │  cd /d/srv/atlas-demo                               │
  │  git pull   ← Git이 GIT_DIR을 읽고 bare repo 기준  │
  │              으로 동작 → 충돌 발생                  │
  └─────────────────────────────────────────────────────┘
```

쉽게 말하면:
> 훅 안에서 `git pull`을 실행하면,  
> 새 폴더로 이동(`cd`)했어도 Git은 "너 지금 bare repo에서 일하고 있잖아" 라고 착각합니다.  
> 왜냐면 `GIT_DIR` 환경변수가 아직 bare repo를 가리키고 있기 때문입니다.

**해결**: 환경변수에 의존하지 않고 `--work-tree`와 `--git-dir`을 **명시적으로 직접 지정**

```bash
# GIT_DIR 환경변수와 무관하게 직접 지정하므로 혼란 없음
git --work-tree="/d/srv/atlas-demo" --git-dir="/c/Users/deploysvc/repos/atlas-demo.git" checkout -f main
```

---

### ❌ 실패 2: pm2 명령어를 찾지 못함

```bash
pm2 restart node/atlas-demo  # ← PATH에 pm2가 없어 "command not found"
```

**원인**: 훅 스크립트는 일반 SSH 로그인 쉘과 다르게 **제한된 환경**에서 실행됩니다. `~/.bashrc`나 `~/.bash_profile`에서 설정한 PATH가 로드되지 않는 경우가 많습니다.

**해결**: pm2 실행 파일의 전체 절대 경로를 사용

```bash
PM2="/c/Users/deploysvc/AppData/Roaming/npm/pm2"
$PM2 restart node/atlas-demo
```

---

### ❌ 실패 3: 실행 권한 누락

```bash
# post-receive 파일을 만들었지만 실행이 안 됨
```

**원인**: Git 훅 파일은 반드시 실행 가능한 파일이어야 합니다. 단순히 파일을 생성하면 기본적으로 실행 권한이 없습니다.

**해결**:
```bash
chmod +x /c/Users/deploysvc/repos/atlas-demo.git/hooks/post-receive
```

---

### ❌ 실패 4: Windows 경로 표기 혼용

```bash
# 잘못된 방법 (Windows 경로를 그대로 사용)
WORK_TREE="D:\srv\atlas-demo"     # ← Git Bash에서 인식 안 됨
GIT_DIR="C:\Users\deploysvc\repos\atlas-demo.git"
```

**원인**: Git Bash는 내부적으로 Unix 파일시스템처럼 동작합니다. Windows의 `\` 경로 대신 `/`를 사용해야 하며, 드라이브 문자도 `/c/`, `/d/`로 표현해야 합니다.

**해결**: Git Bash 스타일 경로로 통일

```bash
WORK_TREE="/d/srv/atlas-demo"
GIT_DIR="/c/Users/deploysvc/repos/atlas-demo.git"
```

## 7. 훅 설계 실전 Best Practices

배포 시스템을 안전하게 구성하기 위한 노하우입니다.

### ✅ 브랜치 보호
항상 배포 대상 브랜치를 명시적으로 필터링하세요. 실험적인 브랜치 push가 실서버에 반영되는 것을 방지합니다.

```bash
if [ "$BRANCH" != "main" ]; then exit 0; fi
```

### ✅ 로그 타임스탬프 출력
배포 시작/완료 시간을 출력해두면 나중에 언제 배포됐는지 추적하기 좋습니다.

```bash
echo ">>> [배포 시작] $(date '+%Y-%m-%d %H:%M:%S')"
```

### ✅ 변경 파일 출력
어떤 파일이 배포됐는지 기록하면 장애 발생 시 원인 추적이 쉽습니다.

```bash
echo "$CHANGED" | sed 's/^/    /'
```

### ✅ 선택적 빌드
매 push마다 전체 빌드를 하면 배포 시간이 길어집니다. 변경된 영역만 빌드하는 것이 효율적입니다.

### ✅ PM2 재시작 최소화
프론트엔드만 변경된 경우 PM2 재시작은 불필요합니다. 불필요한 재시작은 순간적인 서비스 중단을 야기합니다.
