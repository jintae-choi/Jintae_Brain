# Git 브랜치 커밋 복구 가이드

이 문서는 `main` 브랜치에 잘못 커밋한 내용을 원래 작업 브랜치인 `dev`로 옮기고, 푸시 여부를 확인하는 과정에서 사용한 Git 명령어를 정리한 문서입니다.

## 상황

- 원래는 `dev` 브랜치에 커밋하고 푸시해야 했다.
- 실수로 `main` 브랜치에서 커밋했다.
- `main`에는 아직 푸시하지 않았다.
- 잘못 만든 커밋을 `dev`로 옮긴 뒤 `dev`에만 푸시해야 한다.

## 1. 현재 커밋 확인

가장 먼저 `main`에서 방금 만든 커밋 해시를 확인한다.

```bash
git log --oneline
```

### 예시

```bash
$ git log --oneline
56083db fix: update deploy config
2f1c9a1 chore: cleanup env settings
8b7d120 initial setup
```

- 가장 위의 `56083db`가 최근 커밋이다.
- 이 해시를 이후 `cherry-pick`에 사용한다.

## 2. 작업 브랜치로 이동

잘못 커밋한 내용을 옮길 대상 브랜치로 이동한다.

```bash
git checkout dev
```

### 예시

```bash
$ git checkout dev
Switched to branch 'dev'
```

- 이제 이후 작업은 `dev` 브랜치에서 진행된다.

## 3. 잘못 커밋한 내용을 dev 브랜치로 가져오기

`main`에 있는 특정 커밋 하나만 `dev`로 복사한다.

```bash
git cherry-pick <커밋해시>
```

### 예시

```bash
$ git cherry-pick 56083db155bc94319cbd4ab3c4f21cadee07e34c
[dev 91ab234] fix: update deploy config
 Date: Fri Mar 27 10:15:00 2026 +0900
 1 file changed, 12 insertions(+), 2 deletions(-)
```

- `main`의 커밋 내용이 `dev`에 새 커밋으로 복사된다.
- 커밋 메시지는 보통 동일하게 유지된다.

## 4. dev 브랜치를 원격에 푸시

`dev` 브랜치에 복사된 커밋을 원격 저장소로 보낸다.

```bash
git push origin dev
```

### 예시

```bash
$ git push origin dev
Enumerating objects: 8, done.
Counting objects: 100% (8/8), done.
Writing objects: 100% (4/4), 512 bytes | 512.00 KiB/s, done.
To github.com:example/repo.git
   7fa2d10..91ab234  dev -> dev
```

- 이 결과가 나오면 `dev` 브랜치에 정상 푸시된 것이다.

## 5. dev 브랜치에 푸시가 잘 되었는지 확인

원격 `dev` 브랜치의 커밋 목록을 확인한다.

```bash
git log origin/dev --oneline
```

### 예시

```bash
$ git log origin/dev --oneline
91ab234 fix: update deploy config
7fa2d10 feat: add dashboard layout
5ce18ab docs: update readme
```

- 방금 옮긴 커밋이 `origin/dev`에 보이면 원격 푸시는 완료된 상태다.

로컬 `dev` 기준으로도 확인할 수 있다.

```bash
git log dev --oneline
```

## 6. main 브랜치에 푸시되지 않았는지 확인

원격 `main` 브랜치 로그를 확인해서 방금 커밋이 없는지 본다.

```bash
git log origin/main --oneline
```

### 예시

```bash
$ git log origin/main --oneline
2f1c9a1 chore: cleanup env settings
8b7d120 initial setup
```

- `origin/main`에 `56083db` 또는 동일 내용의 커밋이 없으면 `main`에는 아직 푸시되지 않은 것이다.

로컬 `main`도 함께 비교할 수 있다.

```bash
git log main --oneline
```

## 7. main 브랜치의 잘못된 로컬 커밋 제거

`main`에는 푸시하지 않았고, 로컬 커밋만 제거하면 되는 상황이라면 마지막 커밋을 되돌린다.

```bash
git checkout main
git reset --hard HEAD~1
```

### 예시

```bash
$ git checkout main
Switched to branch 'main'

$ git reset --hard HEAD~1
HEAD is now at 2f1c9a1 chore: cleanup env settings
```

- 마지막 잘못된 커밋이 로컬 `main`에서 제거된다.
- `--hard`는 작업 트리까지 함께 되돌리므로, 해당 커밋의 변경사항을 `main`에서 완전히 없애고 싶을 때 사용한다.
- 이미 `dev`에 `cherry-pick`이 끝났는지 먼저 확인한 뒤 실행하는 것이 안전하다.

## 자주 쓰는 확인 명령어

현재 브랜치와 작업 상태 확인:

```bash
git status
```

브랜치별 최근 커밋을 한 번에 비교:

```bash
git log main --oneline
git log dev --oneline
git log origin/main --oneline
git log origin/dev --oneline
```

## 이번 문제 해결 흐름 요약

1. `git log --oneline`으로 `main`의 잘못된 커밋 해시 확인
2. `git checkout dev`로 작업 브랜치 이동
3. `git cherry-pick <커밋해시>`로 커밋 복사
4. `git push origin dev`로 원격 `dev`에 푸시
5. `git log origin/dev --oneline`으로 `dev` 푸시 확인
6. `git log origin/main --oneline`으로 `main` 미푸시 확인
7. 필요하면 `git checkout main` 후 `git reset --hard HEAD~1`로 로컬 `main` 정리

## 주의사항

- `git reset --hard`는 현재 브랜치의 작업 내용까지 제거하므로 신중하게 사용해야 한다.
- `main`에 이미 푸시한 뒤라면 `reset`만으로 끝나지 않고 강제 푸시가 필요할 수 있다.
- 협업 브랜치에서 강제 푸시는 다른 사람 작업에 영향을 줄 수 있으므로 별도 확인이 필요하다.
