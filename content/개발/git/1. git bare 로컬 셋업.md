# Git Bare 저장소 기반 배포 및 협업 환경 구축 가이드

본 문서는 실서버(Windows)를 중앙 원격 저장소로 활용하여 로컬 개발본을 푸시하고, 실서버 웹 디렉토리에 배포 및 동료와 협업하기 위한 전체 과정을 정리한 가이드입니다.

> **💡 구동 환경 안내 (Git Bash 쉘)**
> 본 가이드의 모든 터미널 명령어는 윈도우용 **Git Bash 쉘(Shell)**을 기준으로 작성되었습니다. Git Bash는 Windows 위에서 동작하지만 **리눅스(Linux) 기반의 명령어 생태계**(`ls`, `mkdir`, `/` 형태의 역슬래시 구조 등)를 그대로 사용합니다. 따라서 윈도우 환경임에도 리눅스 서버를 다루듯 일관되고 충돌 없는 명령어 해석을 가능하게 해 줍니다.

## 1. 환경 정보 및 핵심 개념
- **실서버 IP**: 203.0.113.57 (Windows 기반)
- **중앙 Bare 저장소**: `C:\Users\deploysvc\repos\atlas-demo.git`
  - **Bare 저장소란?**: **프로젝트의 "전체 소스 코드"가 100% 빠짐없이 모두 들어있는 것이 맞습니다!** 단지 우리가 눈으로 보고 편집할 수 있는 일반적인 파일 형태(`.html`, `.js` 등)로 풀려 있지 않을 뿐입니다. 눈에 보이는 작업 폴더(워킹 디렉토리) 없이 오직 **압축/암호화된 코드 원본과 모든 버전 관리 데이터**로만 꽉 채워져 있는 "코드 보관 전용 창고"입니다. 사용자가 서버 접속을 통해 직접 코드를 수정하다 꼬이는 것을 원천 차단하기 위해, 오직 코드 백업과 공유(Push/Pull)를 위한 중앙 허브(심장) 역할만 수행합니다. (GitHub 서버 시스템의 핵심 코어 원리와 똑같습니다)
  - **`~/repos` 폴더에 대해**: 깃을 설치하면 자동으로 뿅 하고 생성되는 공간이 아닙니다. 여러 저장소들을 한곳에 깔끔히 모아두기 위해 터미널(`mkdir`)로 **사용자가 명시적으로 직접 만든 폴더**입니다. (`~`는 현재 계정의 홈 디렉토리를 뜻합니다.)
- **실서버 웹 디렉토리**: `D:\srv\atlas-demo`
  - Bare 저장소(보관창고)에서 최신 코드를 당겨와(Fetch) 실제 웹 브라우저가 화면을 띄울 수 있도록 파일들을 진열해놓고 서버를 돌리는 "실제 매장(워킹 디렉토리)" 입니다.
- **로컬 개발 경로**: `D:\workspace\atlas-demo_env`

## 2. 단계별 쉘 명령어 및 상세 해석 해설

### [STEP 1] 실서버: 중앙 Bare 저장소 생성
실서버에 SSH로 접속하여 협업의 중심이 될 빈 저장소(Bare)를 생성합니다.

```bash
# 1. 실서버 터미널에 원격 접속
ssh deploysvc@203.0.113.57

# 2. 지정한 위치에 깃 저장소 관리를 위한 폴더 생성 및 초기화
mkdir -p ~/repos/atlas-demo.git 
cd ~/repos/atlas-demo.git       
git init --bare           
```

### [STEP 2] 로컬 PC: 원격 저장소 등록 및 최초 푸시
로컬 PC에서 개발한 코드를 실서버의 Bare 저장소(보관창고)로 전송(Push)하는 과정입니다. 

```bash
# 1. 로컬 프로젝트 폴더 이동
cd /d/workspace/atlas-demo_env

# 2. 로컬 PC에 연결된 기존 원격 저장소 정보만 끊기 (실제 코드나 원격지에 타격을 주지 않음!)
# "혹시 예전에 작업하느라 다른 서버랑 연결된 주소록(origin)이 있다면 일단 싹 지워줘. 지울 게 없어서 에러가 나도 그냥 휴지통(2>/dev/null)에 몰래 버리고 조용히 넘어가!" 
git remote remove origin 2>/dev/null

# 3. 실서버 Bare 저장소를 origin으로 수동 등록
# "Git아, 저기 있는 저 긴 실서버 주소를, 앞으로 내 코드를 쏘아 올릴 메인 기준점인 'origin'이라는 별명으로 내 주소록에 추가해 줘!"
git remote add origin deploysvc@203.0.113.57:repos/atlas-demo.git

# 4. 서버(보관창고)로 코드 밀어넣기 푸시
# "내 로컬 컴퓨터의 main 브랜치 코드를, 아까 주소록에 적어둔 origin 서버로 쫙 전송(Push)해 줘! 그리고 (-u) 앞으로 이 둘을 딱 연결해놔서 다음번엔 'git push'만 대충 쳐도 알아서 여기가 도착지가 되게 기억해 줘!"
git push -u origin main
```

### [STEP 3] 실서버: 웹 디렉토리 배포 설정 (코드 당겨오기)
실서버(Windows) 안에 다시 접속해서, "보관창고(Bare)"에 있는 최신 코드를 실제 홈페이지가 구동되는 "서비스 폴더(`D:\srv\atlas-demo`)"로 꺼내오는 과정입니다.

> **🚫 왜 간편한 `git clone`을 안 쓰고 이렇게 복잡하게 길게 쓰나요?**
> `git clone`은 강력하지만 **대상이 되는 폴더 안이 100% 텅텅 빈 상태일 때만 작동**합니다.
> 실서버의 웹 디렉토리 환경에는 `.env`(보안 설정)나 `ecosystem.config.cjs`, `nginx` 로그 파일 등 깃에 올리지 않지만 서버 구동에 반드시 필요한 잡동사니 파일들이 이미 존재하고 있을 확률이 매우 높습니다. 이 상태에서 `clone` 명령어를 치면 "빈 폴더가 아니라서 덮어쓰기 무섭다!"라며 에러를 뱉고 뻗어버립니다.
> 따라서, 다음과 같이 4개의 명령어를 쪼개어 수동으로 **"기존에 파일이 들어있던 말던 강제로 덮어씌우는 조립 공정"**을 거치게 됩니다.

```bash
# 1. 웹 서버 서비스 구동 폴더로 이동 (실제 매장)
cd /d/srv/atlas-demo

# 2. 보안 예외 등록 
# "윈도우 다중 계정 문제 때문에, 다른 계정이 이 깃 폴더를 만졌다고 뜨는 소유권 보안 에러를 그냥 눈감아 줘."
git config --global --add safe.directory C:/Users/deploysvc/repos/atlas-demo.git

# 3. 보관창고(Bare)와 강제 동기화 4단 콤보 시전
git init                                     # 1. "일단 이 폴더(안에 뭐가 들어있든)를 Git 추적 폴더로 각성시켜 줘."
git remote add origin /c/Users/deploysvc...    # 2. "이 폴더의 주소록에 아까 만든 Bare 저장소를 메인 서버(origin)로 등록할게."
git fetch origin                             # 3. "아까 등록한 Bare 저장소에 가서 일단 최신 데이터의 '껍데기(압축파일)'만 조용히 가져와서 대기해 줘."
git reset --hard origin/main                 # 4. (핵심) "자, 이제 안에 있던 파일 눈치 보지 말고! 그냥 방금 가져온 코드 상태(origin/main) 그대로 우리 폴더 내용을 무자비하게 강제(hard)로 덮어씌워버려!"
```

### [STEP 4] 동료: 로컬 환경으로 클론
새롭게 프로젝트에 합류한 동료 개발자가 실서버의 코드를 자신의 컴퓨터로 내려받아 협업을 시작합니다. 

```bash
# 동료 PC 터미널에서 수행 (동료는 이미 폴더가 텅 비어있으므로 clone을 쓸 수 있음!)
# "저 실서버 주소에 있는 프로젝트를 통째로 복사해서 내 컴퓨터로 가져와 줘!"
git clone deploysvc@203.0.113.57:repos/atlas-demo.git
```
> **💡 여기서 코드가 다운받아지는 폴더 구조는 어떻게 될까요?**
> 위 명령어를 바탕화면 등 현재 터미널 위치에서 실행했다고 해서, 낱개 파일들이 바탕화면에 더럽게 "콸콸콸" 쏟아지는 것이 **절대 아닙니다**.
> Git은 아주 능지가 높아서, 원격지 저장소의 이름(`atlas-demo.git`)을 보고 알아서 **현재 위치에 `atlas-demo`라는 새로운 이름의 폴더를 하나 예쁘게 먼저 생성합니다.** 그리고 그 방 안에다가 모든 소스 코드를 차곡차곡 다운로드해서 넣어줍니다! 
> (동료들은 폴더에 들어간 뒤 주소록(origin) 설정 할 필요 없이, 바로 `push`와 `pull`을 써서 개발을 시작하면 됩니다.)
